# SAST vs. DAST vs. RASP:对比应用安全测试方法

**来源**: Splunk Blog

构建安全的应用程序是唯一的出路。了解 SAST、DAST 和 RASP 等安全测试解决方案，它们为应用程序提供多层保护。

## 什么是 SAST?
静态应用安全测试 (SAST) 是一种白盒 (white-box) 安全测试方法。SAST 使用应用程序的静态源代码或二进制文件来识别漏洞。这意味着 SAST 工具在应用程序**未运行**时对其代码进行操作。

开发人员使用 SAST 来检测各种安全风险，例如：跨站脚本 (XSS)、不安全的反序列化、缓冲区溢出以及代码中的其他 OWASP 漏洞。由于 SAST 单独使用无法识别运行时特定的漏洞，开发人员通常需要将 SAST 与其他测试方法结合使用，以实现全面的安全性。

SAST 是软件开发生命周期 (SDLC) 的重要组成部分，因为它可以及早发现漏洞。越早发现，修复成本就越低。由于 SAST 工具可以在开发阶段执行，开发人员现在可以在将产品推向市场之前编写代码并对其进行成百上千次的测试 (!!)。

SAST 可以集成到 CI/CD 管道中，这样做时被称为“安全 DevOps”或“DevSecOps”。SAST 通过自动化具有广泛的可扩展性。实施涵盖 SAST 技术的自动化测试的能力，使其成为快速解决代码级风险的高效解决方案。

## 什么是 DAST?
动态应用安全测试 (DAST) 是一种黑盒 (black-box) 测试方法。DAST 中的术语“动态”表明这种方法在应用程序**运行**时评估其安全性。与 SAST 不同，DAST 不需要访问应用程序的源代码。相反，它：
1.  模拟外部攻击者的行为。
2.  从外部测试应用程序，以识别仅在运行时出现的漏洞。

DAST 能够识别各种安全缺陷，如拒绝服务 (DoS) 漏洞和不安全的服务器配置。通过模仿现实世界的攻击场景，DAST 工具评估应用程序如何响应这些模拟威胁。通过这种方法，开发人员可以识别静态测试方法（如 SAST）可能遗漏的弱点。

通常，开发人员在软件开发生命周期的后期阶段（通常就在部署之前）执行 DAST。通过在实时环境中测试应用程序，DAST 提供了对运行时安全性的清晰概览，并支持能够抵御潜在攻击的性能更好的应用程序。

## SAST 是如何工作的？
开发人员使用 SAST 工具 应用预定义的规则和其他检测方法——如模式匹配和数据流分析——来识别编码错误和其他漏洞。他们将这些工具集成到他们的 IDE 或 CI/CD 管道中，以在编码和测试阶段自动扫描。

### 常见的 SAST 方法
以下是 SAST 中使用的一些方法：
*   **模式匹配 (Pattern matching)**：扫描代码库以查找已知的不安全编码实践模式。例如，使用弱加密算法或不安全的 API 调用。
*   **数据流分析 (Data flow analysis)**：跟踪数据如何流经应用程序。这可以通过跟踪从不受信任的输入路径到敏感操作的流向，来识别 SQL 注入或缓冲区溢出等漏洞。
*   **控制流分析 (Control flow analysis)**：分析应用程序的控制结构。例如，循环和条件语句，以发现逻辑缺陷或潜在的漏洞，如竞争条件。
*   **自定义规则创建 (Custom rule creation)**：许多工具允许开发人员创建自定义规则以实施编码最佳实践。这些自定义规则的一些示例包括：
    *   标记 SQL 查询中对任何未清理输入的使用。
    *   识别将用户提供的数据输出到浏览器而没有适当转义的代码。
    *   高亮显示易受攻击或已弃用的函数的使用，例如 JavaScript 中的 `eval()` 或 C 中的 `strcpy()`。
    *   检测源代码中硬编码的 API 密钥和密码，并标记任何缺少用户提供数据输入验证的实例。
*   **依赖项扫描 (Dependency scanning)**：一些 SAST 工具分析应用程序中使用的第三方库和框架，以识别依赖项中的漏洞。
*   **语义分析 (Semantic analysis)**：通过解释代码的含义而不仅仅是结构，SAST 工具可以检测不安全的配置或 API 的滥用。
*   **机器学习模型 (Machine learning models)**：高级 SAST 工具结合机器学习算法，通过分析大型数据集中的模式来识别以前未知的漏洞。

### 使用 SAST 工具时的性能考虑因素
SAST 工具通常需要大量的 CPU 和内存资源。分析大型代码库时，这可能是一个主要问题。对一百万行代码的全量扫描可能会消耗数 GB 的 RAM。
SAST 扫描所需的时间可能会根据代码库的大小而剧烈变化，从小型项目的几分钟到企业级应用程序的数小时不等。这可能会影响：
*   构建管道的时间线
*   开发人员的生产力

考虑使用支持增量扫描的现代 SAST 工具，允许您仅分析更改的代码而不是整个代码库。
此外，通过规则选择和范围定义进行性能调优至关重要。配置不当的 SAST 工具可能会不必要地分析非关键代码路径，浪费时间和资源而没有提供安全效益。

## DAST 是如何工作的？
DAST 工具模拟现实世界的攻击——例如，向输入字段发送各种形式的恶意数据以查看应用程序如何处理它——以识别应用程序行为和响应中的漏洞和弱点。

### DAST 中的步骤
该过程通常涉及以下步骤：
1.  **扫描 (Scanning)**: 扫描 Web 应用程序以发现入口点（如 URL、表单和 API）。此步骤映射出应用程序的结构并识别潜在的攻击面。
2.  **攻击模拟 (Attack simulation)**: 通过向应用程序发送精心设计的请求来模拟恶意活动。这些请求通过尝试利用入口点来测试跨站脚本 (XSS) 和跨站请求伪造 (CSRF) 等漏洞。
3.  **漏洞检测 (Vulnerability detection)**: 分析应用程序的响应以识别安全弱点。它评估应用程序在攻击下是否按预期运行。例如，可以注入恶意数据以检测 SQL 注入缺陷。
4.  **报告 (Reporting)**: 生成包含检测到的漏洞和修复建议的报告。开发人员可以使用报告中的结果来修复已识别的问题。

现代 DAST 解决方案可以包含高级功能，如 AI 驱动的分析和实时数据集成。它们自动创建测试集。动态适应应用程序的结构，并使用机器学习算法最大限度地减少误报。

### 使用 DAST 工具时的性能考虑因素
在使用 DAST 工具时，您需要牢记某些性能瓶颈：
*   DAST 工具主动与运行中的应用程序交互，这会暂时增加应用程序和 Web 服务器上的 CPU 和内存使用率。
*   DAST 扫描期间产生的大量并发请求可能会消耗大量的网络带宽。
*   DAST 扫描器通常通过生成唯一请求来绕过或使应用程序缓存失效，从而降低缓存机制的有效性。
*   DAST 工具创建和管理多个测试会话可能会增加应用程序服务器上的内存使用率。

为了避免此类问题或尽可能减少其影响，请遵循以下做法：
*   在应用程序使用率最低的非高峰时段安排 DAST 扫描。
*   在镜像生产设置的预发布或 staging 环境中进行扫描。
*   使用 DAST 工具中的节流功能来控制并发请求的数量。
*   在扫描期间增加应用程序服务器线程池。
*   为 DAST 扫描器 IP 配置单独的缓存策略。
*   如果超过性能阈值，实施自动扫描暂停。

## SAST vs. DAST: 关键区别

此表总结了 SAST 和 DAST 之间的区别：

| 特性 | SAST (静态) | DAST (动态) |
| :--- | :--- | :--- |
| **测试类型** | 白盒 | 黑盒 |
| **分析对象** | 源代码、字节码、二进制 | 运行中的应用程序 |
| **需要运行应用?** | 否 | 是 |
| **发现时间** | 开发早期 (SDLC Early) | 开发后期 (SDLC Late) |
| **典型漏洞** | 代码注入, 弱加密, 缓冲区溢出 | 运行时错误, 配置错误, 认证问题 |
| **误报率** | 较高 (缺乏运行时上下文) | 较低 (验证了实际攻击) |
| **语言依赖性** | 强 (需要支持特定语言) | 弱 (语言无关) |
| **成本** | 修复成本低 | 修复成本高 |

## SAST 和 DAST 的优缺点

### SAST 的优点
*   在软件开发生命周期的早期识别安全漏洞。
*   分析代码库，包括基本功能和复杂的分支。
*   直接在源代码上操作，无需运行应用程序。
*   提供实时反馈和全面的见解，例如漏洞的确切位置。
*   生成可导出的报告，可以使用仪表板进行跟踪。
*   支持自动化，因此比手动代码审查更快、更高效。

### SAST 的缺点
*   通常会产生大量的误报。因此，它可能需要手动审查。
*   无法识别运行时或特定于环境的漏洞，例如配置错误。
*   需要特定于每种编程语言的工具，因此维护很复杂。
*   难以理解外部库、API 和 REST 端点。

### DAST 的优点
*   识别仅在应用程序执行期间出现的漏洞。
*   在外部操作，不需要访问源代码。这对测试第三方应用程序或编译代码很有效。
*   模拟实际攻击场景并识别 SAST 遗漏的风险。
*   独立于应用程序的编程语言工作。
*   评估整个应用程序和系统。这包括内存消耗、资源使用和第三方接口。

### DAST 的缺点
*   仅关注应用程序的外层，可能会遗漏深层的漏洞。
*   在 SDLC 后期应用，因此修复更昂贵且耗时。
*   遗漏某些 SAST 工具可以通过源代码分析识别的漏洞。例如，考虑一个 Web 应用程序使用不安全的随机数生成器（如 JavaScript 中的 `Math.random()`）来创建会话令牌。DAST 可能会遗漏此缺陷，因为它只测试运行时行为。（但 SAST 可以检测到它。）
*   对于大型项目需要自定义基础设施，并且并行运行多个应用程序实例需要大量资源。

## 我应该何时使用 SAST 对比 DAST?

### 何时使用 SAST
*   在早期开发阶段：一旦编写了代码就实施 SAST。这有助于在它们深深嵌入代码库之前捕获漏洞。
*   为了合规性：许多行业法规（如 [PCI-DSS](https://www.splunk.com/en_us/blog/learn/pci-dss-compliance.html) 和 HIPAA）要求对源代码进行安全审查。
*   在 CI/CD 管道中：自动化 SAST 扫描以确保持续安全性，而不会减慢开发速度。
*   用于代码质量保证：除了安全性之外，SAST 还可以帮助强制执行编码标准和最佳实践。

### 何时使用 DAST
*   在预生产/Staging 环境中：在部署之前运行 DAST 以捕获运行时问题和配置错误。
*   对于遗留应用程序：当源代码不可用或过时时，DAST 是识别漏洞的唯一方法。
*   验证补丁：使用 DAST 验证安全修复是否有效并且没有引入新的问题。
*   为了渗透测试：DAST 模拟外部攻击者的视角，使其成为[渗透测试](https://www.splunk.com/en_us/blog/learn/penetration-testing.html)的重要工具。

## 哪个更有效：SAST 还是 DAST?
答案并非二选一。最有效的安全策略可以利用这两种方法的优势。SAST 提供了深度的代码级洞察，而 DAST 提供了真实世界的运行时视角。通过结合两者，您可以实现全面的安全覆盖。

### 采取混合方法 (IAST)
交互式应用安全测试 (IAST) 结合了 SAST 和 DAST 的元素。IAST 代理在应用程序运行时在应用程序内部运行（比如说就在 JVM 中），分析代码执行和数据流以实时识别漏洞。

### 将 RASP 用作 SAST 和 DAST 的替代方案
运行时应用程序自我保护 (RASP) 是一种安全技术，它构建在应用程序或其运行时环境中。它能够在应用程序运行时检测并阻止实时攻击。与扫描漏洞的 SAST 和 DAST 不同，RASP 主动保护应用程序免受利用。

## 应用程序安全测试的未来是 SAST、DAST 和 RASP

随着网络威胁的不断演变，仅依赖一种安全测试方法已不再足够。未来在于集成 SAST、DAST 和 RASP 的整体方法。这种多层策略确保在每个阶段都能识别和缓解漏洞——从编码和测试到部署和运行时。
尤其是对于金融服务 (FinServ) 行业以及涉及 IoT、边缘计算和连接设备的场景，这种分层防御至关重要。

## 弹性安全始于主动测试
在当今的数字环境中，被动安全性已不再是一种选择。组织必须采取主动立场，在开发过程的早期集成强大的安全测试。通过利用 SAST、DAST 和 RASP 的力量，您可以构建不仅安全而且能够抵御不断变化的威胁环境的弹性应用程序。
