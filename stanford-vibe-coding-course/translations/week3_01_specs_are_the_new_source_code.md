# 规范已死，规范万岁！(Specs Are the New Source Code)

**作者**: Ravi Mehta
**日期**: 2025年

**核心观点**: 原型是新的规范，而规范是新的……源代码？

在我的职业生涯中，规范（Specs）变得越来越短。当我在微软开始工作时，产品经理（PM）的价值是用他们规范的重量来衡量的。多年后，在 Tripadvisor，我们神话了一位带着写在餐巾纸上的规范去参加产品评审的 PM。

产品团队通常将规范视为文书工作——在交付的“真正工作”之前的必要之恶。工程师因优雅的代码而受到赞誉，设计师因漂亮的用户体验而受到赞誉。PM 因交付影响而获得荣誉。但是规范？它们通常被匆忙完成，被扔在一边，被遗忘。

这以前一直是个错误。但在今天，我们构建的方式正在发生转变。工程师变得越来越快——快得多。AI 可以在几分钟内将粗略的想法变成可工作的代码。瓶颈不再是构建。瓶颈是**知道要构建什么**，并让团队围绕这些需求保持一致。

突然间，不起眼的规范不再是短暂的文书工作。它是产品管理的基础——它正在成为源代码本身。

## 为什么 PM 突然成了瓶颈
Andrew Ng 在他最近的演讲 [Building Faster with AI](https://www.youtube.com/watch?v=RNJCfif1dPY) 中指出了一个前所未有的趋势：
> “我有生以来第一次看到经理向我提议配备两倍于工程师数量的 PM。我仍然不知道这个提议是否是个好主意，但我认为这是世界走向的一个标志。”

随着工程师使用 AI 交付的速度提高数倍，公司需要更多的 PM 来支持这些高效的工程师，而不是更少。这给产品管理的各个方面带来了巨大的压力——理解客户需求、精心制作正确的功能、验证影响。所有这些压力都集中在一个工件上——**规范**。

## 规范——新的源代码
在传统软件开发中，程序员编写人类可读的“源代码”，然后将其编译成高度优化的、机器可读的“目标代码”（二进制文件）。源代码实际上是事实来源（Source of Truth）。

OpenAI 的 Sean Grove 有一个挑衅性的论点。在他最近的演讲 [The New Code](https://www.youtube.com/watch?v=8rABwKRsec4) 中，他认为**写得好的 Prompt（即规范）是新的源代码**。

从这个角度看，我们做 AI 开发的方式是倒退的。我们精心制作 Prompt 向模型传达我们的意图。AI 生成代码。然后我们保留代码并丢弃 Prompt。
Grove 观察到：“这感觉就像你粉碎了源代码，然后非常小心地对二进制文件进行版本控制。”

在传统编程中，源代码是神圣的。源代码包含注释、结构和文档——理解和修改系统所需的一切。二进制文件只是下游的产物。
但在 AI 时代，我们翻转了这种关系。我们将生成的代码视为值得保留的工件，而将规范——Prompt——视为一次性的。

Grove 认为这完全搞反了。代码，即使是优雅的代码，也是他所说的规范的“有损投影”。就像反编译二进制文件不会给你原始的注释和变量名一样，阅读代码也不会告诉你背后的全部意图。

**规范包含了一切。** 一个足够健壮的规范可以生成“好的 TypeScript、好的 Rust、服务器、客户端、文档、教程、博客文章，甚至播客。”
更重要的是，规范做了代码做不到的事情：它们在共同目标上使人类和机器保持一致。

> “在不久的将来，沟通最有效的人就是最有价值的程序员。字面上看，如果你能有效地沟通，你就能编程。” —— Sean Grove

新的稀缺技能不是编码。而是编写完全捕捉意图和价值观的规范。

## 规范驱动开发 (Spec-Driven Development) 实战
Danny Martinez（Decimals 创始人）展示了他的“规范驱动开发”流程。这个流程产生了两个影响：
1. 大大改善了与工程部门关于大功能的沟通。
2. 使 Danny 能够在没有编码经验的情况下，从详细规范一直做到上线小功能。

### 案例：Danny 的流程
**任务**: 在落地页头部添加一个按钮，链接到申请页面。
**工具**: Linear (项目管理), VS Code, GitHub Copilot Pro, Claude Sonnet 4, Linear MCP server。

**步骤**:
1. **生成 Ticket**: 从 Slack 消息中生成 Linear Ticket。
2. **澄清需求**: 在 Ticket 中明确想要的新文案。
3. **Prompt Claude**: 打开 Copilot，提示 Claude 打开 Linear Ticket。
4. **分析**: 提示 Claude 审查 Ticket 并相对于代码库进行分析。
5. **实施**: 提示 Claude 创建分支并实施这些更改。
6. **测试**: 验证更改是否符合预期。
7. **PR**: 打开 GitHub PR。
8. **审查**: 等待工程师批准。

这个流程的核心在于，**通过规范（Linear Ticket/Prompt）来驱动代码生成**，而不是直接跳进代码里。

## 结论
规范不再是死板的文档，它是**活的、可执行的意图**。在这个新世界里，能够清晰、准确地编写规范（无论是 PRD 还是 Prompt）将成为区分优秀产品经理和开发者的关键能力。

## 关键术语

| 英文 | 中文 | 说明 |
|------|------|------|
| Source of Truth | 事实来源 | 系统中最权威的信息源，传统上是源代码，现在正变为规范 |
| Lossy Projection | 有损投影 | 从规范到代码的转换过程中，原始意图（如注释、背景）的丢失 |
| Spec-Driven Development | 规范驱动开发 | 以编写详细规范为核心，由 AI 负责生成的开发模式 |
| Object Code / Binary | 目标代码/二进制 | 编译后的机器代码，文中比喻为 AI 生成的代码 |
